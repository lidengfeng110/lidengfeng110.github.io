实验 ： 搭建mysql 数据库的主从复制

先在一台服务器上安装上mariadb 服务器，用来做主服务器，先来修改配置文件                vim /etc/my.cnf

接着重启数据库服务器
systemctl restart mariadb
重启后就会在数据库的目录 /var/lib/mysql/ 下生成相应的二进制文件
如： master-bin.000001 , master-bin.index

接下来创建账号及口令并授权账号复制权限。
grant replication slave on . to repluser@’192.168.30.%’ identified by ‘centos’;

用命令查看一下二进制位置，记住这个位置，将来复制时知道从哪复制。
show master logs;

然后给数据库导入自己的数据，如： mysql < hellodb_innodb.sql
这样在主服务器上的操作就做完了。

接下来就在从服务器上进行配置。
先安装上 mariadb 服务器并且启动，然后在配置文件中进行修改 vim /etc/my.cnf
在里面添加 server_id=2 # 这块一定不能跟主服务器一样，不然会冲突。

然后重启服务 systemctl restart mariadb

接下来在mysql 中执行命令，这个命令比较长，不要紧，可以查看帮助，然后复制过来就行。
help change master to
在里面把我们需要的复制下来就可以了，然后在这基础上修改就行。

接下来就可以进行复制了，复制前可以先拿命令查看下从服务器的状态
show slave status\G
复制： start slave
复制完后再看下状态就会发现数据发生变化。并且把主服务器上 hellodb 数据库就同步过来。

现在在次验证下同步过程，先在主服务器上创建一个数据库
create database db1;
然后在从服务器上用命令查看会发现已经同步。

现在又发生变化了，主服务器又对数据库进行修改，但是修改时从服务器断电了，等从服务器重启后，主服务器也修改完成了。 试想从服务器启动后数据会自动同步吗？
实验证明从服务器重启后查看数据时数据已经同步。并且 I/O 线程和 SQL 线程自动启动。

刚才主服务器跟从服务器都是干净的系统，但一般在生产中有很多都是刚开始只有一个服务器，到后来访问量上涨，又要求搭建从服务器，那么这样后加的服务器数据怎么同步？（这样也使用于扩容从服务器）

首先看主服务器的配置文件，因为刚开始只有一台服务器的时候，配置文件中没有加 server_id=1 , innodb_file_per_table 这些配置，但是现在添加服务器了所以要加上，还有二进制日志也要加上。
加上后重启服务 systemctl restart mariadb 

然后在从服务器上的配置文件中也添加上  server_id =2    
     加完后启动服务         systemctl start mariadb

因为现在这个从服务器是个干净的数据库，接下来怎么让主从进行同步呢？
首先不可能让从直接去同步主服务器上的数据，因为如果主服务器上原来的数据过大，那么直接同步不太现实，应该是先把主服务器上的数据做下备份，然后导入到从服务器上，最后在让主从进行同步，应该是按照这样的思路进行解决。 

先在主服务器上创建一个账号

mysql > grant replication slave on . to repluser@’192.168.30.%’ identified by ‘centos’;

现在在主服务器上进行备份
mysqldump -A -F –single-transaction –master-data=1 > all.sql

备份完后然后把脚本复制到从服务器上
scp all.sql 192.168.39.27:

接下来在从服务器上还不能直接拿这个脚本进行恢复，要先打开这个脚本，在里面进行修改。
vim all.sql

接下来把脚本导入数据库即可
mysql < all.sql
这样就把主服务器上的数据导入到从服务器上了，接下来就可以做同步操作了
只用在从服务器上的数据库中执行 start slave ; 就可以同步了。 

但是现在还有个缺点，同步是同步了，但是在从服务器上也可以对数据进行修改，这是不安全的，所以基于安全，建议在从服务器的配置文件中添加  read_only 就可以了，这样从服务器就不能修改了。

到此就实现了事先有一个主服务器，后来怎么添加从服务器的过程。

如果主节点已经运行了一段时间，且有大量数据时，如何配置并启动slave节点
通过备份恢复数据至从服务器
复制起始位置为备份时，二进制日志文件及其POS
如果要启用级联复制,需要在从服务器启用以下配置
[mysqld]
log_bin
log_slave_updates

复制架构中应该注意的问题：
1、限制从服务器为只读
在从服务器上设置read_only=ON
注意：此限制对拥有SUPER权限的用户均无效
阻止所有用户, 包括主服务器复制的更新
mysql> FLUSH TABLES WITH READ LOCK;（这种方法不合理，不用做）

2、如何保证主从复制的事务安全
参看https://mariadb.com/kb/en/library/server-system-variables/
在master节点启用参数：
sync_binlog=1 每次写后立即同步二进制日志到磁盘，性能差

要是以性能为主 写0 ， 以安全为主，写1 。

如果用到的为InnoDB存储引擎：
innodb_flush_logs_at_trx_commit=1 每次事务提交立即同步 日志写磁盘
innodb_support_xa=ON 默认值，分布式事务MariaDB10.3.0废除
sync_master_info=# #次事件后master.info同步到磁盘
在slave节点启用服务器选项
–skip_slave_start=ON 不自动启动slave
在slave节点启用参数：
sync_relay_log=# #次写后同步relay log到磁盘
sync_relay_log_info=# #次事务后同步relay-log.info到磁盘

实验：实现了数据库主从复制的级联复制

一般我们数据库的主从复制都是一个主，多个从这样来进行配置，但是主服务器即要接受用户的增删改，还要负责把修改的二进制日志复制到从服务器上，这就加大了主服务器的压力，那么现在有一种方案来解决这种问题，就是在主服务器跟从服务器的中间添加一个专门用来复制的从服务器，来缓解主服务器的压力，这种操作就叫级联复制。 

现在我们就来做一次级联复制的实验。首先我们最少要准备三台主机来做实验。

三台主机分别是 ： 192.168.30.17 ， 192.168.30.27 ， 192.168.30.37 

 用 192.168.30.17 来做主服务器  
然后对主服务器的配置文件进行修改      vim /etc/my.cnf

修改完后重启服务 systemctl restart mariadb
这样主服务器配置就初步完成了
接下来要创建一个账号并授权
grant replication slave on . to repluser@’192.168.30.%’ identified by ‘centos’;
可以用命令来查看下二进制日志位置 show master logs; # 现在是400

接下来用 192.168.30.27 这台主机来做中间的那台从服务器。
首先修改配置文件 vim /etc/my.cnf
在里面添加 log_bin # 因为作为中间的服务器，将来要把二进制日志同步到别的从服务器上去，所以必须启动。
server_id=2
log_slave_updates # 添加这条变量可以记录主服务器上同步过来的更新，从而记录到自己的二进制日志中去。

然后在数据库中执行命令：
C

执行完这个命令可以拿命令先查看下从服务器的状态： show slave status\G
然后进行同步 start slave;
如果复制的过程中发现信息有错误，想从头来重新复制，可以用命令 ： reset slave all; 清除复制
还要添加一个账号
grant replication slave on . to repluser@’192.168.30.%’ identified by ‘centos’;
这个账号是用来将来复制数据到其他从服务器上的。
